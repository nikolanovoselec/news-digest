# Architecture

System overview, component map, and data flow.

**Audience:** Developers

---

## Overview

news-digest is a single Cloudflare Worker serving an Astro-rendered app. Every 4 hours (00/04/08/12/16/20 UTC), a Cron Trigger fires the global-feed coordinator: it fans out 63 curated RSS/Atom/JSON sources, canonical-URL-deduplicates candidates, and enqueues chunked LLM summarization jobs to a Cloudflare Queue. Chunk consumers write articles to a shared D1 pool and, on the last chunk of each tick, enqueue a single finalize message that runs one additional LLM call to collapse same-story pairs that landed in different chunks (cross-publisher dedup). Per-user dashboards read from that pool filtered by the user's active hashtags — no per-user LLM calls. A 5-minute cron drains pending tag discoveries and dispatches daily digest emails to opted-in users at their configured local time. A daily cron at 03:00 UTC purges articles older than 14 days (starred articles are exempt). See [`sdd/README.md`](../sdd/README.md) for product intent.

Implements [REQ-PIPE-001](../sdd/generation.md#req-pipe-001-global-scrape-and-summarise-pipeline-on-a-fixed-cadence), [REQ-PIPE-008](../sdd/generation.md#req-pipe-008-cross-chunk-semantic-dedup-pass).

## Components

| Component | Role |
|---|---|
| Astro Worker (request handler) | Serves all HTML pages and JSON APIs; runs in the Cloudflare Workers runtime |
| Cron Trigger | 5-minute trigger — sweeper + discovery + scheduled-digest dispatcher |
| Queue Consumer | Processes `SCRAPE_COORDINATOR`, `SCRAPE_CHUNKS`, and `SCRAPE_FINALIZE` messages; each message runs in its own isolate |
| D1 | Strongly-consistent storage for users, digests, articles, pending_discoveries |
| KV | Edge-distributed cache for discovered sources, headlines, source health |
| Workers AI | LLM inference for digest summarization, source discovery, and cross-chunk dedup finalize pass |
| Resend | Transactional email for "digest ready" notifications |
| Federated OAuth (GitHub, Google) | Sign-in via configured providers; at least one provider pair must be set |

## Request Lifecycle

### Global-feed pipeline (every 4 hours)

Implements [REQ-PIPE-001](../sdd/generation.md#req-pipe-001-global-scrape-and-summarise-pipeline-on-a-fixed-cadence), [REQ-PIPE-002](../sdd/generation.md#req-pipe-002-chunked-llm-processing-with-json-output-contract), [REQ-PIPE-008](../sdd/generation.md#req-pipe-008-cross-chunk-semantic-dedup-pass).

```
Cron fires (every 4 hours: 00/04/08/12/16/20 UTC)
  → SCRAPE_COORDINATOR queue message sent
Coordinator consumer
  → fans out all {tag × curated-source} pairs (concurrency cap: 10)
  → per-source fetch outcome recorded in KV: source_health:{url}
      success → deletes counter; failure → increments counter (7-day TTL)
  → per-source fetch failure logged; never propagates (resilient fan-out)
  → feed snippets extracted: <content:encoded> → <description> → HN story_text → Reddit selftext
  → eviction pass (REQ-DISC-003): discovered-tag URLs at 30 consecutive failures
      → removed from sources:{tag} KV entry; per-URL health counter cleared
      → if tag's feed list is now empty → INSERT pending_discoveries (user_id='__system__')
         so the discovery cron repopulates it on the next 5-minute invocation
      → curated URLs tracked in the health counter but never evicted at runtime
  → candidates older than 48 h (by parsed publish date) are dropped; candidates with no parsable date are kept
  → canonical-URL deduplication across all candidates
  → chunks ~100 candidates → enqueues one SCRAPE_CHUNK message per chunk
Chunk consumer (per chunk, isolated)
  → fetches article HTML bodies for candidates with snippet < 400 chars (20-worker concurrency)
  → merges fetched body with feed snippet — takes whichever is longer
  → single Workers AI call (JSON output; default model: gpt-oss-20b)
  → LLM enforces 150–200-word contract on the details field (≥ 120 words required): 2 or 3 paragraphs (WHAT happened / HOW it works / optionally IMPACT for the reader), each 3–5 sentences, \n-separated; each article echoes its input index so the consumer aligns by index (falls back to positional when < 3 articles echo a valid index or < 50% echo); title-overlap sanity check drops any article whose LLM-generated title shares no substantive token with the source headline
  → LLM-cluster + canonical-URL dedupe (first-source-wins)
  → env.DB.batch([articles, alternative_sources, tags, scrape_run counters])
  → chunk failure marks only that chunk failed; other chunks persist
  → completion gate (CF-002 — atomic, race-free):
      → INSERT OR IGNORE INTO scrape_chunk_completions (idempotent under redelivery)
      → SELECT COUNT(*) FROM scrape_chunk_completions WHERE scrape_run_id = ?
      → if count >= total_chunks: UPDATE scrape_runs SET finalize_enqueued = 1
          WHERE id = ? AND finalize_enqueued = 0 (exactly one consumer wins; others see meta.changes = 0)
      → winning consumer stamps run status = ready (articles become visible immediately)
          and enqueues one SCRAPE_FINALIZE message carrying the scrape_run_id
          finishRun uses COALESCE(finished_at, now) — redelivered messages preserve the
          original finish time so the history page duration does not drift forward (REQ-PIPE-006 AC 6)
          finishRun UPDATE is guarded WHERE status = 'running': once a run leaves running
          its status is terminal — a late-arriving failed chunk cannot flap the dashboard
          back to failed, and a late success cannot flip a failed run back to ready (REQ-PIPE-006 AC 4)
      → addChunkStats is gated on isFirstCompletion (meta.changes === 1):
          redelivered chunks skip the additive stats UPDATE so token, cost, and
          article counters advance exactly once per chunk (REQ-PIPE-006 AC 7)
      → if SCRAPE_FINALIZE.send fails: roll back finalize_enqueued to 0 so the next
          queue redelivery re-acquires the gate and retries the handoff (REQ-PIPE-008 AC 9)
          rollback is itself guarded — if D1 is also down the failure is logged as
          digest.generation{status:finalize_lock_rollback_failed} and the send error is rethrown
      → KV counter (scrape_run:{id}:chunks_remaining) maintained as a derived mirror
          for /api/scrape-status progress display only — not the completion gate
Finalize consumer (one message per tick, REQ-PIPE-008)
  → loads up to 250 surviving articles for the run (ingested_at DESC; tail skipped when cap applies)
  → skips LLM call when ≤ 1 article (finalize_noop)
  → single Workers AI call over title + source + pub-ts list (FINALIZE_DEDUP_SYSTEM prompt)
      → requests dedup_groups: index pairs for cross-publisher same-story pairs
      → primary model; falls back to secondary if primary returns malformed JSON
  → for each dedup group (size ≥ 2):
      → winner = earliest published_at; ascending id tiebreaker for determinism
      → loser merge: 6-statement D1 batch per loser
          (a) loser primary source → winner alt-source row
          (b) loser existing alt sources re-pointed to winner
          (c) tag union (loser tags inserted into winner's tag set)
          (d) user stars re-pointed to winner (preserves read marks across merge)
          (e) user reads re-pointed to winner
          (f) loser article row deleted (CASCADE cleans remaining child rows)
      → all INSERTs use OR IGNORE — idempotent on message redelivery
  → folds tokens + cost + losers_deleted into addChunkStats (gated on losersDeleted > 0 to avoid double-counting on retry)
  → on permanent failure: logs finalize_failed, leaves articles un-merged; run stays ready
  Known limitation: ticks that produced > 250 articles skip cross-chunk dedup on the tail (token budget cap)
```

### Daily retention cleanup

```
Cron fires (daily at 03:00 UTC)
  → Pass 1 — article retention (REQ-PIPE-005):
      DELETE articles WHERE published_at < now-14d AND NOT starred by any user
      FK cascade removes alternative sources, tag rows, read-tracking rows
      Note: scrape_chunk_completions rows are NOT cleaned by this pass —
        cleanup.ts does not yet implement that sweep (migration 0007 comment
        describes the intended steady-state; table can grow until implemented)
  → Pass 2 — stuck-tag prune (REQ-DISC-006):
      Scan sources:{tag} KV entries — for each entry with feeds:[] AND
        discovered_at < now-7d → remove that tag from every user's hashtags_json
      Clears permanent "Stuck tag" warning on /settings for tags the LLM
        could never find sources for
  → Pass 3 — orphan-tag KV sweep (REQ-PIPE-007):
      DELETE sources:{tag} and discovery_failures:{tag} for every tag

      (picks up entries left unowned by Pass 2 in the same cron run)
```

## Data Flow

Articles are the central entity in the global pool. Each article belongs to a `scrape_runs` tick, not a user. Users read from the pool by filtering on their active hashtags. Foreign keys cascade on delete. Starred articles are user-scoped and exempt from the 14-day cleanup.

Pending discoveries are per-user rows but discovery results (`sources:{tag}` KV) are globally shared so multiple users benefit from a single discovery run.

The `pending_discoveries` table is also written by the system when a feed eviction empties a tag's source list. These rows carry `user_id = '__system__'` (a synthetic sentinel that is never a real ULID) so queries scoped to a real user naturally exclude them. The discovery cron groups by tag and picks up both real-user and system-owned rows on the next invocation.

The `/settings` page surfaces a "Stuck tags" fieldset ([REQ-DISC-004](../sdd/discovery.md#req-disc-004-manual-re-discover)) when any of the session user's tags has no entry in the curated source registry AND its `sources:{tag}` KV entry is absent, empty, or unparseable. A single "Discover missing sources" button in that fieldset posts to `POST /api/admin/discovery/retry-bulk` — via a native HTML form so the flow works in Samsung Browser and in-app webviews — which clears the stale KV entries and re-queues a discovery pass for every empty-feeds tag in one D1 batch. The route is gated by Cloudflare Access (`/api/admin/*` wildcard); non-admin users see the Access login prompt rather than the re-queue succeeding. The single-tag JSON variant `POST /api/admin/discovery/retry` is preserved for scripted callers.
