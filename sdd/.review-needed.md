## REQ-SET-006 AC 5 / REQ-SET-007 AC 7 — missing dedicated 429-on-exhaustion tests (MEDIUM)

The two acceptance criteria added on 2026-04-29 — discovery-progress polling rate-limit (REQ-SET-006 AC 5) and timezone-update rate-limit (REQ-SET-007 AC 7) — wire `enforceRateLimit` into the corresponding endpoints, but the existing test files exercise the rate-limit branch only structurally (the in-memory KV stub keeps the limiter from throwing — neither file asserts on a 429 response after the bucket is drained).

- **Contract that's missing a test**: each endpoint must return HTTP 429 with `Retry-After` once a single user exhausts their per-minute bucket (30/min for `SET_TZ`, 120/min for `DISCOVERY_STATUS`).
- **Where the gap lives**: `tests/auth/set-tz.test.ts` and `tests/settings/discovery-status.test.ts`. Both have the KV stub already; what's missing is a test that pre-seeds the bucket past the limit and asserts on `response.status === 429` and the presence of a `Retry-After` header.
- **Why this is escalated, not auto-fixed**: writing tests is `tdd-guide`'s lane. Spec-reviewer flags the gap; the user invokes `tdd-guide` to add the cases.
- **Severity**: MEDIUM — the rate-limit wiring itself is exercised end-to-end by the structural tests, so the limiter is not silently fail-opening; what's missing is a regression guard against a future change that bypasses it.

## REQ-AUTH-002 AC 1 / REQ-AUTH-003 AC 3 — cookie-attribute mechanism leakage (MEDIUM, JUDGMENT)

The mechanism-leakage rule in `spec-discipline.md` lists cookie attributes (`HttpOnly`, `SameSite=Lax`, `Secure`, `Path=/`, `__Host-` prefix) as belonging in `documentation/security.md`, not in the AC of a REQ. REQ-AUTH-002 AC 1 enumerates them inline:

> "...a short-lived **access cookie** (HMAC-SHA256 JWT, 5-minute TTL, `__Host-` prefix, `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`)..."

REQ-AUTH-003 AC 3 also mentions `SameSite=Lax` parenthetically.

- **Why this is escalated, not auto-rewritten**: the user-observable rewrite ("JavaScript on the page cannot read the session token; cross-site GETs do not carry the session") is a substantial reshape that risks losing the security contract. AND the mechanism description has nowhere to land — `documentation/security.md` does not exist in this project (current docs: README, api-reference, architecture, configuration, deployment, decisions/). Either:
  1. Create `documentation/security.md` first (doc-updater's lane, not spec-reviewer's), then rewrite the REQs to backlink, OR
  2. Decide that for the Authentication domain, cookie attributes ARE the contract (similar to the existing user-override for `email_enabled` — the column name IS the contract because it's the shared identifier across UI, DB, and dispatcher), in which case add an entry to `sdd/.user-overrides.md`.
- **Severity**: MEDIUM — present since 2026-04-23, has been through three review passes without being flagged, and the security review on PR #159 did not call it out either. Reasonable case for option (2).
