## 2026-05-05 — fix/article-yield-recovery PR review

### MEDIUM — Two parallel Google News paths now exist; ownership is unclear

Two independent code paths now produce a Google News query-RSS source for the same tag:

1. **Discovery-LLM path** (REQ-DISC-001 AC 3): for tags without a curated source, the LLM-discovery prompt instructs the model to include a Google News fallback URL, which is validated and persisted to KV `sources:{tag}` with no TTL. This is per-tag, written once at discovery time.
2. **Coordinator-baseline path** (new REQ-PIPE-001 AC 9): every tick, the coordinator synthesises a per-tag GN query-RSS source for every tag in (defaults ∪ curated ∪ discovered KV). The skip set is bespoke `google-news-*` curated entries only — it does NOT skip tags whose KV `sources:{tag}` already contains a GN URL written by the discovery LLM.

Net effect: a discovered non-curated tag without a first-party feed will fan out a GN query twice per tick (once via the KV-cached discovery URL, once via coordinator synthesis). The query strings differ slightly (LLM-crafted vs. tag-with-dashes-as-spaces), so canonical-URL dedup may not catch every overlap; the prefer-direct-source pass cleans up downstream but only when a direct copy lands in the same tick.

This is not a spec conflict — both ACs read correctly and describe distinct mechanisms. But it is an **unresolved architectural question**:

- Should REQ-DISC-001 AC 3 drop the GN fallback now that REQ-PIPE-001 AC 9 provides per-tag GN baseline at the coordinator? The discovery LLM could focus solely on first-party sources, and the coordinator owns GN entirely.
- Or should REQ-PIPE-001 AC 9's skip set widen to cover tags whose KV cache already contains any GN URL?

Resolution path: record an ADR in `documentation/decisions/` (likely titled "Google News baseline lives at the coordinator, not in discovery") that picks one ownership model. Until then both paths coexist with mild redundancy that the dedup pass mostly absorbs.

### MEDIUM — Per-item source-name labelling has no AC home (changelog drift)

The 2026-05-05 changelog entry "REQ-PIPE-001 source-name labelling sharpened" describes a user-observable behavior change: the read-at-source picker now displays the underlying publisher (e.g. "Help Net Security") for Google News items instead of the feed-level adapter label ("Google News: mcp"). No AC of REQ-PIPE-001 was modified in this PR. By the strict letter of the changelog-drift rule the entry is drift, but deletion is the wrong fix — the underlying behavior IS user-observable and worth pinning in the spec. The right fix is adding an AC; that placement is a JUDGMENT call:

- **Option A (REQ-PIPE-001 AC 4 amendment):** AC 4 already governs the multi-source append rule on re-discovery. Extend it with a sentence: "When a feed entry carries a per-item publisher name (e.g. an RSS `<source>` element on Google News items), that name is used for the source label in the article's source list rather than the feed-level adapter name; absent or empty per-item names fall back to the feed-level name." This keeps the rule with the source-list construction REQ.
- **Option B (REQ-READ-005 AC 5 amendment):** AC 5 currently says "each source's name and per-source timestamp" without specifying whose name. Extend it with: "The source name is the underlying publisher when the originating feed identified one per-item; otherwise it is the feed-level source name." Recommendation: Option A. The label is determined at ingestion time (sources.ts), not at render time (reading surface). The reading REQ should keep saying "each source's name" generically and let the pipeline REQ define what that name resolves to.

Conservative resolution (unleashed): escalated rather than auto-applied because rewriting AC 4 inline mixes two distinct rules (multi-source append + per-item label resolution) and risks obscuring the existing first-ingestion-timestamp guarantee. User picks the wording.

### MEDIUM — REQ-PIPE-008 AC 1 silent on the "different studies don't merge" rule

The prompt change in `src/lib/prompts.ts` adds explicit anti-merge guidance: studies, audits, and benchmarks with different numbers, methodology, or authors are NOT same-story pairs and must never be merged (the worked example uses two MCP-vulnerability studies citing 25% and 6.2%). The change is reachable to users — they will now see two separate cards for studies with conflicting numbers instead of risking a false merge that hides one finding.

REQ-PIPE-008 AC 1 currently says the dedup pass identifies "same-story pairs by their actual content rather than by surface lexical overlap." This is compatible with the new prompt rule but silent on the specific judgment criterion. A future prompt regression that re-introduces study-merging would not be caught against the spec because the spec does not name the anti-rule.

Two related issues travel together here:

1. **Missing AC tightening.** AC 1 should call out that two studies, audits, or benchmarks on the same topic citing different numbers, methodology, or authors are distinct stories and not eligible for merging — even when their bodies discuss identical subject matter.
2. **Missing changelog entry.** No 2026-05-05 entry covers the prompt tightening. Per the changelog discipline this counts as a behavioral change worth a user-facing line ("the cross-chunk dedup pass no longer merges studies/benchmarks with different numbers or methodology, so conflicting findings on the same topic stay as separate cards").

Conservative resolution (unleashed): escalated rather than auto-applied because the precise scope of the anti-rule is an intent decision (studies/audits/benchmarks only? or any "different numbers" content including price comparisons, performance benchmarks, election polls?). Auto-rewriting AC 1 with the narrower wording would lock in a scope decision without the user weighing in.
